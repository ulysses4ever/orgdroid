# M10 — Undo / Redo

## Goal
Add a safety net for the ~13 destructive operations the app has accumulated by M9. Two icon buttons appear in the top bar:

- **Undo** reverses the last content-changing action (delete, indent, outdent, move up/down, TODO cycle, title commit, notes commit, priority change, tag add/remove, sibling-add, zoom).
- **Redo** re-applies an undone action — until the user makes a new content change, which clears the redo stack.

Stack depth is bounded at 50. Both stacks are cleared on Save, Discard-local, and Open. The implementation is pure: each public mutating method captures a snapshot before applying its change, and `undo()`/`redo()` swap snapshots between the two stacks while also clearing any in-flight edit buffer. No async work; no parser changes; no Compose work beyond two new IconButtons.

After M10, every destructive operation is reversible. This unblocks confident exploration of the outline ops and is a prerequisite for any later "destructive batch" work (move-many, delete-many, etc.).

## What we keep from M9
- All M1–M9 behavior unchanged: SAF open/save, outline + collapse + zoom, inline title/notes editing, TODO cycle, indent/outdent, move up/down, conflict detection, recents + auto-open + back-to-recents, search/filter, metadata sheet.
- `OrgParser`, `OrgSerializer`, `Node`, `NodeId`, `TreeOps`, `FileIo`, `RecentFiles`, `Search`, `Theme`, `MainActivity` — unchanged.
- `OutlineState` gains two fields (`undoStack`, `redoStack`) — both default empty so nothing breaks on construction.
- `OutlineRow` is unchanged. `MetadataSheet` is unchanged. `SearchBarRow` is unchanged.

## Out of scope for M10 (don't implement)
- **Per-keystroke undo inside the title/notes editor.** The buffer is not snapshotted; clicking Undo while editing CLEARS the buffer outright. User loses pending text. This matches our "title/notes is buffer-only until commit" model.
- **Coalescing.** Two indents in a row are two undo steps, not one. No timing-based merging.
- **Undo for `toggleCollapse` / `beginEdit` / `openSearch` / `openMetadata` / `setSearchQuery` / `cancelEdit`.** These mutate UI state only (or implicitly commit, in which case the commit's own push covers it). Pure UI state is not undoable.
- **Redo across a Save.** Save clears both stacks.
- **Undo across `discardLocal` or `open(uri)`.** The new tree has fresh NodeIds; old snapshots would dangle. Both clear stacks.
- **Persisted undo history.** Stacks live in the ViewModel only; lost on process death (same as collapse state today).
- **Custom undo limit per-file.** Single constant `UndoOps.LIMIT = 50`.
- **Keyboard shortcut** (Ctrl-Z, Cmd-Z). No hardware-keyboard story yet.
- **Visual hint on disabled Undo button** beyond Material's default grayed state.
- **Undo for `removeRecent` / `closeFile` / `cancelClose`.** Recents is a separate persistent store; close resets state entirely. Both out of undo scope.
- **Restoring metadata sheet on undo.** `metadataSheetFor` is cleared by `undo()`/`redo()`; if you undo back to a state where the sheet target still exists, the sheet does NOT reopen.
- **Snackbar "Undone: deleted X"** notifications. The visual change in the outline IS the feedback.
- **Coalescing tag adds.** Adding `:foo:` then `:bar:` is two steps.
- **Bundling implicit commit + explicit op as one step.** Typing in row X then tapping Delete on row Y is two undos. Acceptable; see Pitfall 4.

## File contents

### 1. `app/src/main/kotlin/dev/orgdroid/outline/Undo.kt` (new)

```kotlin
package dev.orgdroid.outline

import dev.orgdroid.org.Node
import dev.orgdroid.org.NodeId

/**
 * One reversible point in the document's history.
 *
 *   - [root]: the parsed tree at this point.
 *   - [collapsed]: which subtrees were collapsed.
 *   - [focusedRoot]: which heading was zoomed-in, or null.
 *   - [dirty]: whether the document was dirty (unsaved) at this point.
 *
 * Transient UI state (edit buffers, search query, metadata sheet, IO flags) is
 * intentionally NOT captured: undo restores the document, not the in-flight UI.
 */
data class UndoSnapshot(
    val root: Node,
    val collapsed: Set<NodeId>,
    val focusedRoot: NodeId?,
    val dirty: Boolean,
)

object UndoOps {
    const val LIMIT = 50

    /**
     * Append [snap] to [stack], capping at [LIMIT]. If [snap] equals the
     * current top, returns [stack] unchanged (dedupe — prevents rapid clicks
     * or no-op mutations from bloating the history with identical entries).
     */
    fun push(stack: List<UndoSnapshot>, snap: UndoSnapshot): List<UndoSnapshot> {
        if (stack.lastOrNull() == snap) return stack
        return (stack + snap).takeLast(LIMIT)
    }
}
```

### 2. `app/src/test/kotlin/dev/orgdroid/outline/UndoTest.kt` (new)

```kotlin
package dev.orgdroid.outline

import dev.orgdroid.org.OrgParser
import org.junit.Assert.assertEquals
import org.junit.Assert.assertSame
import org.junit.Test

class UndoTest {

    private fun snap(text: String): UndoSnapshot {
        val root = OrgParser.parse(text)
        return UndoSnapshot(
            root = root,
            collapsed = emptySet(),
            focusedRoot = null,
            dirty = false,
        )
    }

    @Test fun pushAddsToEnd() {
        val a = snap("* A\n")
        val b = snap("* B\n")
        val stack = UndoOps.push(emptyList(), a)
        val next = UndoOps.push(stack, b)
        assertEquals(listOf(a, b), next)
    }

    @Test fun pushDedupesIdenticalTop() {
        // Use the same snapshot instance twice — don't depend on parser ID
        // stability across parses (OrgParser doesn't guarantee it).
        val a = snap("* A\n")
        val stack = UndoOps.push(emptyList(), a)
        val next = UndoOps.push(stack, a)
        assertSame(stack, next)
    }

    @Test fun pushDifferentSnapshotsAreNotDeduped() {
        val a = snap("* A\n")
        val b = snap("* B\n")
        val stack = UndoOps.push(UndoOps.push(emptyList(), a), b)
        assertEquals(2, stack.size)
    }

    @Test fun pushCapsAtLimit() {
        var stack = emptyList<UndoSnapshot>()
        repeat(UndoOps.LIMIT + 5) { i ->
            stack = UndoOps.push(stack, snap("* H$i\n"))
        }
        assertEquals(UndoOps.LIMIT, stack.size)
        // Oldest dropped — first surviving entry corresponds to index 5
        assertEquals("H5", stack.first().root.children.single().title)
    }

    @Test fun pushDropsOldestOnOverflow() {
        var stack = emptyList<UndoSnapshot>()
        repeat(UndoOps.LIMIT) { i -> stack = UndoOps.push(stack, snap("* H$i\n")) }
        val newest = snap("* NEW\n")
        stack = UndoOps.push(stack, newest)
        assertEquals(UndoOps.LIMIT, stack.size)
        assertEquals("H1", stack.first().root.children.single().title)
        assertEquals(newest, stack.last())
    }
}
```

**3b. Add a private extension function** near the top of the `OutlineViewModel` class body (just below `private val store = …`):

```kotlin
private fun OutlineState.withSnapshot(): OutlineState {
    val r = root ?: return this
    val snap = UndoSnapshot(r, collapsed, focusedRoot, dirty)
    return copy(
        undoStack = UndoOps.push(undoStack, snap),
        redoStack = emptyList(),
    )
}
```

This is the entire undo-recording primitive. It:
- Captures the receiver state's `(root, collapsed, focusedRoot, dirty)` as a snapshot.
- Appends to `undoStack` via `UndoOps.push` (which dedupes and caps).
- Clears `redoStack` — any new mutation invalidates the redo history.

Usage: every public mutating method's final `_state.value = s.copy(…)` becomes `_state.value = s.withSnapshot().copy(…)`. The snapshot captures `s`'s pre-mutation content; the `.copy(…)` overlays the mutation. Final state has BOTH the snapshot recorded AND the mutation applied.

**3c. Add `undo()` and `redo()` methods.** Place them near the existing `closeMetadata` / `setPriority` group (or wherever feels logical — they're entry points for the new top-bar buttons):

```kotlin
fun undo() {
    val s = _state.value
    val snap = s.undoStack.lastOrNull() ?: return
    val currentRoot = s.root ?: return
    val currentSnap = UndoSnapshot(currentRoot, s.collapsed, s.focusedRoot, s.dirty)
    _state.value = s.copy(
        root = snap.root,
        collapsed = snap.collapsed,
        focusedRoot = snap.focusedRoot,
        dirty = snap.dirty,
        undoStack = s.undoStack.dropLast(1),
        redoStack = (s.redoStack + currentSnap).takeLast(UndoOps.LIMIT),
        editing = null,
        editBuffer = "",
        editingNotes = null,
        notesBuffer = "",
        metadataSheetFor = null,
    )
}

fun redo() {
    val s = _state.value
    val snap = s.redoStack.lastOrNull() ?: return
    val currentRoot = s.root ?: return
    val currentSnap = UndoSnapshot(currentRoot, s.collapsed, s.focusedRoot, s.dirty)
    _state.value = s.copy(
        root = snap.root,
        collapsed = snap.collapsed,
        focusedRoot = snap.focusedRoot,
        dirty = snap.dirty,
        undoStack = (s.undoStack + currentSnap).takeLast(UndoOps.LIMIT),
        redoStack = s.redoStack.dropLast(1),
        editing = null,
        editBuffer = "",
        editingNotes = null,
        notesBuffer = "",
        metadataSheetFor = null,
```

Both methods:
- Early-return when their stack is empty (so the button being enabled is a sufficient pre-condition).
- Push the current state onto the OPPOSITE stack (undo → redo, redo → undo).
- Restore `(root, collapsed, focusedRoot, dirty)` from the popped snapshot.
- Clear edit buffers and the metadata sheet target. Pending typing is lost — this is by design (see Pitfall 1).

**3d. Patch `commitEditInternal`** to push a snapshot whenever a commit changes content. Find the existing two `if (node != null && ...)` blocks. Inside each, wrap the mutation with `.withSnapshot()`:

```kotlin
private fun commitEditInternal(s: OutlineState): OutlineState {
    var current = s
    val editingId = current.editing
    if (editingId != null) {
        val root = current.root
        if (root != null) {
            val node = TreeOps.findNode(root, editingId)
            if (node != null && node.title != current.editBuffer) {
                current = current.withSnapshot().copy(
                    root = TreeOps.updateTitle(current.root!!, editingId, current.editBuffer),
                    dirty = true,
                )
            }
        }
        current = current.copy(editing = null, editBuffer = "")
    }
    val notesId = current.editingNotes
    if (notesId != null) {
        val root = current.root
        if (root != null) {
            val node = TreeOps.findNode(root, notesId)
            if (node != null) {
                val newNotes = if (current.notesBuffer.isEmpty()) emptyList()

                if (newNotes != node.notes) {
                    current = current.withSnapshot().copy(
                        root = TreeOps.updateNotes(current.root!!, notesId, newNotes),
                        dirty = true,

                }
            }

        current = current.copy(editingNotes = null, notesBuffer = "")
```

## Pitfalls (read before coding)

1. **Undo CLEARS edit buffers.** A user mid-typing in row X who clicks Undo loses their typed text. The snapshot doesn't include `editBuffer`/`notesBuffer`, and `undo()` explicitly null-clears `editing`/`editingNotes`. Document in the eventual README. The alternative (commit-then-undo) would push two undo steps for one click, which is worse.

2. **`withSnapshot()` MUST be on the pre-mutation state.** The pattern is `s.withSnapshot().copy(root = newRoot, …)`. If you write `s.copy(root = newRoot, …).withSnapshot()`, the snapshot captures the POST-mutation state and undo becomes a no-op. The chain order is load-bearing.

3. **Early returns don't pollute the stack.** Because the snapshot is added inside the final `_state.value = …` assignment, methods that `return` before that assignment never persist the snapshot. The `if (newRoot === root) return` patterns in `moveUp` / `moveDown` / `indent` / `outdent` / `cycleTodo` are correct as-is.

4. **Implicit commit + explicit op = two undo steps.** Type "foo" into row X, then tap Delete on row Y. Result: undoStack has two entries (one for the commit, one for the delete). Two Undo clicks fully revert. This is consistent — each atomic change is its own undo step — but worth knowing during manual testing.

5. **Dedupe is by data-class equality.** Two snapshots with `equal` root references AND equal `collapsed`/`focusedRoot`/`dirty` are deduped at the top of the stack only. This stops rapid-clicking from inflating history; it does NOT stop e.g. "make change, undo, make same change" from creating distinct entries (the redo-stack clearing on push ensures correctness regardless).

6. **Clear stacks on Save AND Discard.** Forgetting either leaves stale snapshots that reference NodeIds (Discard case) or pre-save content (Save case) that would confuse users. Don't conflate these with `open(uri)` — open already constructs a fresh state.

7. **Memory is bounded but not zero.** 50 snapshots × shared-Node tree references = O(touched-nodes per mutation × 50). Structural sharing via immutable `Node.copy()` keeps unchanged subtrees shared. For a 10k-heading file, this stays under ~1 MB. Don't bump `LIMIT` past 100 without measuring.

8. **`metadataSheetFor` is wiped by undo/redo.** Cleanest semantics: undo restores the document; sheet UI is the user's reopening. If `metadataSheetFor` survived undo and the target node was deleted in the popped snapshot, the sheet would already auto-hide via `findNode == null`, but having a non-null pointer at a gone node is awkward.

9. **`searchActive` / `searchQuery` survive undo.** Search is a filter overlaying the document; restoring the document re-derives the filter naturally. Don't clear them in `undo()` / `redo()`. (`metadataSheetFor` is different because the sheet renders the target node directly.)

10. **Don't push undo for `toggleCollapse`.** Collapse is navigation; cluttering history with collapse toggles would let collapse spam push out real content history (50-cap is shared). The snapshot still captures `collapsed` so other ops correctly preserve the collapse context, but pure collapse toggling does not create undo entries. Same logic for `setSearchQuery`, `closeMetadata`, etc.

11. **Icons must be `AutoMirrored`.** `Icons.Filled.Undo` and `Icons.Filled.Redo` are deprecated in favor of `Icons.AutoMirrored.Filled.{Undo,Redo}` since Compose Material 1.6. Using the auto-mirrored versions avoids new warnings and supports RTL.

12. **TopBar real estate.** After M10, actions are `[Undo][Redo][Search][Close][Open][Save]` — 4 icons + 2 text buttons. On a 360dp screen the title may truncate. Acceptable; deferred-polish flag in TODO already covers grouping/overflow.

13. **`UndoSnapshot` uses `data class` equality.** `Node` is also a `data class` so the auto-generated `equals` recurses through `children`. For the dedupe check at the top of `UndoOps.push`, this is O(N) per push in the worst case (when comparing two snapshots of the same tree). In practice the top-of-stack snapshot rarely equals the next one (mutations change root), so the recursive equals short-circuits at the root or first level. Don't worry unless profiling shows it.

14. **No `OutlineRow` signature change.** Undo lives entirely in the ViewModel + top bar. Rows don't get an Undo callback. If you find yourself editing `OutlineRow`, you took a wrong turn.

15. **`UndoOps.LIMIT = 50` is a top-level constant.** Don't inline as a magic number in `OutlineViewModel`. The constant is referenced in three places: `UndoOps.push`, `undo()`'s redo-cap, and `redo()`'s undo-cap.

16. **Disabled-but-visible buttons.** `IconButton(enabled = false)` shows the icon at reduced opacity. Don't conditionally hide them — the gray state IS the discovery affordance.

17. **`closeFile()` triggers `performClose` which constructs a fresh `OutlineState(recents = …)`** — both stacks default to empty. No explicit clear needed. Same for `confirmCloseDiscard`. Both correct as-is.

18. **The 21st `OutlineState` field count is now 23.** Existing TODO.md note about the overdue split should be re-counted. Update the index when M10 lands.

19. **No test for `undo()` / `redo()` directly.** These are AndroidViewModel methods; testing requires Robolectric or a refactor. The pure `UndoOps.push` is the only thing covered by `UndoTest`. Acceptable — the orchestration is small and exercised end-to-end by the manual acceptance criteria.

## What M10 explicitly does NOT do

- Per-keystroke undo / intra-edit history.
- Coalescing adjacent same-type ops into one undo step.
- Undo across Save, Discard, or Open.
- Persisted undo history (across process death).
- Keyboard shortcuts (Ctrl-Z, Cmd-Z).
- Undo for `toggleCollapse`, `setSearchQuery`, `openMetadata`, `openSearch`, or any UI-state-only mutator.
- Restoring `metadataSheetFor` or `editing`/`editingNotes` on undo.
- Snackbar / toast notifications on undo/redo.
- Configurable stack limit.
- A refactor of `OutlineState` into split scopes (still deferred to a future milestone).
- Conversion of Open/Save TextButtons to icons to free up top-bar space.
- Compose UI tests for the new buttons.
