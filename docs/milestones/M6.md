# M6 — Move up / Move down siblings

## Goal
Close the last gap in basic outline manipulation. Indent and outdent (M4) change a node's parent; this milestone adds the orthogonal axis: change a node's position **within its current parent**. After M6, a user can fully reorder an outline without manually retyping headings.

The whole milestone is one TreeOps pair (`moveUp`/`moveDown`), two ViewModel methods, two menu items, and seven tests. No new dependencies. No new files.

## What we keep from M5
- All M1–M5 behavior: SAF open/save, outline rendering, inline title editing, conflict detection, indent/outdent, zoom in/out, FAB, TODO cycle, notes editing.
- `OrgParser`, `OrgSerializer`, `Node`, `NodeId`, `FileIo`, `MainActivity`, `Theme` — unchanged.
- All existing TreeOps untouched. We only **add** functions.

## Out of scope for M6 (don't implement)
- Drag-to-reorder (visual handles, long-press-drag-release). Adds significant Compose complexity; M9 candidate at the earliest.
- Move across parents (i.e., "promote then move"). User can chain outdent + move themselves.
- "Move to top" / "Move to bottom" — only single-step moves.
- Animated row swaps. `LazyColumn` re-renders are fine.
- Keyboard shortcuts (no hardware-keyboard story yet).
- Reordering the root (no top-level concept of "root sibling" exists; root is synthetic).
- Editing priority or tags — still deferred to the metadata sheet.
- A "Move ▸" submenu to declutter the dropdown. Inline both items; submenu support in Material 3 needs custom layout work. Track in TODO.md.

## Acceptance criteria (must all pass)
- [ ] `./gradlew test` passes; new tests in `TreeOpsTest` and `OrgSerializerTest` green.
- [ ] `./gradlew assembleDebug` succeeds with no new warnings.
- [ ] Open `test.org`. Long-press the second top-level heading → menu shows "Move up" (enabled) and "Move down" (enabled if there's a third heading; else greyed). Tap "Move up" → the heading swaps positions with the first.
- [ ] Long-press the FIRST top-level heading → "Move up" is disabled; "Move down" is enabled.
- [ ] Long-press the LAST top-level heading → "Move down" is disabled; "Move up" is enabled.
- [ ] Move a heading that has children → entire subtree moves with it (children preserved, levels unchanged).
- [ ] Move a heading whose title was just edited but not yet committed → the title commits as part of the move (same pattern as M4's indent/outdent / M5's cycleTodo).
- [ ] Save. `adb pull` and diff: the moved heading's body and children are byte-identical, just at a new position. Unmoved headings are byte-identical to before (no `rawHeadingLine` invalidation).
- [ ] Zoom into a parent, then long-press one of its visible children → "Move up"/"Move down" enabled per position **within the focused parent**. Move works; the focused-root view does not break.
- [ ] Rotate the device after a move → tree state preserved.

## File contents

### 1. `app/src/main/kotlin/dev/orgdroid/org/TreeOps.kt` (additions only)

Add the following inside the existing `object TreeOps`, **after** the existing `outdent` function and **before** the M5 `updateTodoState`. Do NOT remove or rename any existing function.

```kotlin
/**
 * Swap [id] with its previous sibling within the same parent. No-op if
 * [id] is the first child of its parent (no previous sibling), is the
 * synthetic root, or is not found. Returns [root] unchanged on no-op so
 * callers can use referential equality (`newRoot === root`) to short-circuit.
 *
 * Move does NOT change the node's level, children, notes, or
 * [Node.rawHeadingLine]. Round-trip is byte-identical for the moved
 * subtree.
 */
fun moveUp(root: Node, id: NodeId): Node {
    val (parent, idx) = findParentAndIndex(root, id) ?: return root
    if (idx == 0) return root
    val newChildren = parent.children.toMutableList().apply {
        val node = removeAt(idx)
        add(idx - 1, node)
    }
    return transform(root, parent.id) { it.copy(children = newChildren) }
}

/**
 * Swap [id] with its next sibling within the same parent. No-op if [id]
 * is the last child of its parent, is the synthetic root, or is not found.
 * Same round-trip guarantees as [moveUp].
 */
fun moveDown(root: Node, id: NodeId): Node {
    val (parent, idx) = findParentAndIndex(root, id) ?: return root
    if (idx >= parent.children.size - 1) return root
    val newChildren = parent.children.toMutableList().apply {
        val node = removeAt(idx)
        add(idx + 1, node)
    }
    return transform(root, parent.id) { it.copy(children = newChildren) }
}
```

Implementer notes:
- These functions are deliberately symmetric. Do NOT try to factor through a single `swapSiblings(parent, i, j)` helper — the asymmetry of the no-op guard (`idx == 0` vs `idx >= size - 1`) is easier to read inline.
- The `transform(root, parent.id) { it.copy(children = newChildren) }` call rebuilds only the path from root to the affected parent. Unaffected branches share references (the immutability invariant the rest of TreeOps depends on).
- `rawHeadingLine` is **not** cleared. This is load-bearing for the byte-identical round-trip test.
- `findParentAndIndex` returns `null` for `root.id` (root is not its own child) and for unknown ids, so both edge cases collapse into "return root unchanged".

### 4. `app/src/main/kotlin/dev/orgdroid/outline/OutlineViewModel.kt` (modify in place)

Add the following two methods near `indent` and `outdent`. They follow the same pattern: commit any pending edit, attempt the move, short-circuit on no-op via referential equality.

```kotlin
fun moveUp(id: NodeId) {
    val s = commitEditInternal(_state.value)
    val root = s.root ?: return
    val newRoot = TreeOps.moveUp(root, id)
    if (newRoot === root) return
    _state.value = s.copy(root = newRoot, dirty = true)
}

fun moveDown(id: NodeId) {
    val s = commitEditInternal(_state.value)
    val root = s.root ?: return
    val newRoot = TreeOps.moveDown(root, id)
    if (newRoot === root) return
    _state.value = s.copy(root = newRoot, dirty = true)
}
```

Nothing else in the ViewModel changes. `commitEditInternal` (already generalized in M5 to flush both title and notes) handles edit hygiene transparently.

**5b. Update `flatten`** — the `visit` inner function adds two fields to each emitted `RenderItem`:

```kotlin
private fun flatten(
    root: Node?,
    collapsed: Set<NodeId>,
    focusedRoot: NodeId?,
): List<RenderItem> {
    if (root == null) return emptyList()
    val startNode = if (focusedRoot != null) TreeOps.findNode(root, focusedRoot) else root
    if (startNode == null) return emptyList()

    val out = mutableListOf<RenderItem>()
    fun visit(parent: Node, node: Node, indexInParent: Int, depth: Int) {
        val isCollapsed = node.id in collapsed
        if (node.id != startNode.id) {
            out.add(
                RenderItem(
                    node = node,
                    depth = depth,
                    hasChildren = node.children.isNotEmpty(),
                    isCollapsed = isCollapsed,
                    canIndent = indexInParent > 0,
                    canOutdent = parent.id != startNode.id,
                    canMoveUp = indexInParent > 0,
                    canMoveDown = indexInParent < parent.children.size - 1,
                )
            )
        }
        if (!isCollapsed) {
            val childDepth = if (node.id == startNode.id) 0 else depth + 1
            for ((i, c) in node.children.withIndex()) visit(node, c, i, childDepth)
        }
    }
    visit(startNode, startNode, -1, 0)
    return out
}
```

Two things to notice:
- `canMoveUp == canIndent` mathematically (both are "have a previous sibling"). They are kept as separate fields because they answer **semantically distinct** UI questions ("can I demote this?" vs "can I shift this up?"). Don't merge them — the next milestone could change either independently.
- The focused-root case (`parent == startNode`) does not affect `canMoveUp/canMoveDown`: movement stays within the parent, so the ceiling check that `canOutdent` needs is irrelevant here.

## Pitfalls (read before coding)

1. **Do NOT clear `rawHeadingLine`.** Move-up/move-down change the *position* of a node in its parent's children list. The node itself — heading text, todo state, priority, tags, notes — is identical. If you accidentally clear `rawHeadingLine` (e.g. by routing through `bumpLevels` or `updateTitle`), the serializer re-builds the heading from parsed fields and tests will catch the byte-drift via `moveUpRoundTripsByteIdenticalForMovedSubtree`.
2. **Do NOT touch level.** Moving within a parent never changes depth. Resist any urge to call `bumpLevels`.
3. **`canMoveUp == canIndent` mathematically** but keep them as separate fields. They are conceptually independent and will diverge if the indent rule ever gains a constraint (e.g. "can't indent the first child of a focused root").
4. **`canMoveDown` requires `parent.children.size`** which `flatten`'s `visit` already has access to via the for-loop, but the per-emit decision uses `indexInParent < parent.children.size - 1`. Don't compute `lastIndexOf` — the index is right there.
5. **Move + focused-root.** When zoomed in on a parent, you see only its descendants. The parent itself isn't rendered, so the user can't accidentally move the focused root. The children can still be reordered freely **within** the focused parent. No special handling needed in `moveUp/moveDown`.
6. **Commit pending edits FIRST** (mirror `indent`/`outdent`/`cycleTodo`). `OutlineViewModel.moveUp` starts with `val s = commitEditInternal(_state.value)` so a half-typed title or notes edit isn't dropped on the floor.
7. **Referential-equality short-circuit.** Both `moveUp` and `moveDown` return the same `root` instance on no-op (first sibling for up, last sibling for down). The ViewModel detects this with `newRoot === root` and skips the state update; this prevents needless recomposition and prevents the `dirty` flag from flipping on a no-op tap.
8. **Don't add a `swapSiblings` helper.** It's a tempting DRY target but only saves two lines and obscures the asymmetric no-op guards.
9. **Test fixture sharing.** Use the existing `tree()` helper at the top of `TreeOpsTest`. Don't redefine the fixture in each test.
10. **Menu order matters.** Place Move up / Move down between Outdent and New sibling. Keeping movement (indent, outdent, move-up, move-down) clustered helps users build muscle memory before we eventually move to a submenu.
11. **No animation.** `LazyColumn` will reshuffle the rows because their `NodeId` keys haven't changed — Compose's animation system will not animate the move by default, but that's fine for M6. A future polish item.
12. **`OutlineRow` is approaching 25 parameters.** That's painful but the alternative — a refactor to bundle callbacks — is its own milestone. Don't try to slip it into M6.
