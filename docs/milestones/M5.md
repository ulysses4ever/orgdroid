# M5 — Field editing: TODO cycle + notes editing

## Goal
Close the biggest "I can see it but can't touch it" gap left after M4. Two features that together complete the basic editability of a heading:

1. **TODO chip cycle.** Tap on an existing TODO/DONE chip → cycle to the next state. Add a "Toggle TODO" item to the long-press menu so users can cycle even from no-state (since no chip is shown then). The cycle is `null → TODO → DONE → null`.
2. **Notes editing.** Tap on the notes block under a heading → enter a multi-line text field for the body lines. Add an "Edit notes" item to the long-press menu so the editor works even when notes are empty. Commit on tap-away (same pattern as title edit).

That's the whole milestone. No new fields beyond TODO state and notes; priority and tags stay read-only.

## What we keep from M4
- `OrgParser`, `OrgSerializer`, `Node`, `NodeId`, `FileIo`, `MainActivity`, `Theme`.
- All existing TreeOps (`findNode`, `transform`, `updateTitle`, `insertSiblingAfter`, `appendTopLevel`, `delete`, `maxNodeIdValue`, `findParentAndIndex`, `indent`, `outdent`).
- M3 inline title editing + save-with-conflict-detection.
- M4 indent/outdent + zoom in/out + context-aware FAB.

## Out of scope for M5 (don't implement)
- Editing priority — defer to M6/M7 (metadata sheet).
- Editing tags — defer to M6/M7.
- Adding a TODO via tapping a "+ TODO" placeholder chip — only via the menu, to avoid visual clutter.
- Custom `#+TODO:` workflow keywords — keep the hardcoded `[null, "TODO", "DONE"]` cycle.
- Live preview of notes structure (lists, formatting). Notes are a plain monospace editor.
- Multi-line title editing. Title stays single-line.
- "Undo" for a TODO cycle or notes edit.
- A separate "Done" button to commit the notes editor. Commit happens implicitly on tap-away or any other state-changing action.
- Validating that notes don't accidentally introduce headings (lines starting with `*`) or break drawers. Document as a footgun (see Pitfalls).
- Move up / move down. M6.

### 1. `app/src/main/kotlin/dev/orgdroid/org/TreeOps.kt` (additions only)

Add the following inside the existing `object TreeOps`. Do **not** remove or rename any existing function.

```kotlin
/**
 * Replace the TODO keyword of [id]. Pass null to remove the keyword. Clears
 * [Node.rawHeadingLine] so the serializer reconstructs the heading.
 */
fun updateTodoState(root: Node, id: NodeId, newState: String?): Node =
    transform(root, id) { it.copy(todoState = newState, rawHeadingLine = null) }

/**
 * Cycle the TODO state: null → "TODO" → "DONE" → null. This is the only
 * mutation surface for TODO state in M5; future milestones may take a
 * configurable keyword list.
 */
fun cycleTodoState(root: Node, id: NodeId): Node {
    val node = findNode(root, id) ?: return root
    val next = when (node.todoState) {
        null -> "TODO"
        "TODO" -> "DONE"
        "DONE" -> null
        else -> null  // unknown keyword → clear, then user can cycle again
    }
    return updateTodoState(root, id, next)
}

/**
 * Replace the notes (body lines) of [id]. Does NOT clear [Node.rawHeadingLine]
 * because the heading line is unaffected. The serializer emits notes directly
 * via emitLines(); they round-trip verbatim.
 */
fun updateNotes(root: Node, id: NodeId, newNotes: List<String>): Node =
    transform(root, id) { it.copy(notes = newNotes) }
```

Notes for the implementer:
- `cycleTodoState` reads the current state via `findNode` then dispatches. Don't try to inline the cycle into a `transform` lambda — you'd lose the ability to short-circuit on "node not found".
- `updateTodoState` clears `rawHeadingLine`. `updateNotes` does NOT. This asymmetry is load-bearing.
- The `else -> null` branch in `cycleTodoState` is defensive: if a future config introduces e.g. `WAITING`, encountering it cycles back to null, allowing the user to recover.

```kotlin
private fun commitEditInternal(s: OutlineState): OutlineState {
    var current = s
    // 1. Commit title edit if active.
    val editingId = current.editing
    if (editingId != null) {
        val root = current.root
        if (root != null) {
            val node = TreeOps.findNode(root, editingId)
            if (node != null && node.title != current.editBuffer) {
                current = current.copy(
                    root = TreeOps.updateTitle(root, editingId, current.editBuffer),
                    dirty = true,
                )
            }
        }
        current = current.copy(editing = null, editBuffer = "")
    }
    // 2. Commit notes edit if active.
    val notesId = current.editingNotes
    if (notesId != null) {
        val root = current.root
        if (root != null) {
            val node = TreeOps.findNode(root, notesId)
            if (node != null) {
                val newNotes = if (current.notesBuffer.isEmpty()) emptyList()
                               else current.notesBuffer.split('\n')
                if (newNotes != node.notes) {
                    current = current.copy(
                        root = TreeOps.updateNotes(root, notesId, newNotes),
                        dirty = true,
                    )
                }
            }
        }
        current = current.copy(editingNotes = null, notesBuffer = "")
    }
    return current
}
```

## Pitfalls (read before coding)

1. **`updateNotes` does NOT clear `rawHeadingLine`. `updateTodoState` DOES.** This asymmetry is critical for round-trip stability: editing the body should not invalidate the heading line; editing the TODO keyword must, because the keyword sits inside the heading string. The `updateNotesPreservesRawHeading` and `todoStateChangeReconstructsHeading` tests catch this.
2. **`commitEditInternal` now commits TWO things.** Title first, then notes. Both can be active simultaneously is impossible (`beginEdit*` flushes the other first), but the function still handles each independently so reordering / refactoring is safer.
3. **`notes.split('\n')` for the buffer-to-list conversion.** Empty buffer → `emptyList()`, not `[""]`. A buffer of `"a\n\nb"` → `["a", "", "b"]`. Don't use `split('\n').filter { it.isNotEmpty() }` — that destroys intentional blank lines in the body.
4. **`BasicTextField` for notes must have `singleLine = false`** (the default — just don't set it to true). Setting `keyboardOptions = KeyboardOptions(imeAction = ImeAction.Done)` would replace the Enter key with a Done action and prevent newline insertion. For notes, DON'T set imeAction; use the default `ImeAction.Default` which lets the system show Enter as newline.
5. **`TodoChip`'s `clickable` must be on the `Surface`, not the inner `Text`.** Tapping the small padding area around the text counts as a hit on the Surface. Placing `clickable` inside the Text would leave the padding dead.
6. **Click propagation: chip vs row.** `TitleRow` has `combinedClickable`. `TodoChip` has its own `clickable`. Compose's gesture detection assigns the tap to the deepest interactive descendant — the chip — so tapping the chip does NOT also fire the row's title-edit. Manual test: tapping the chip should cycle the TODO state and NOT enter title-edit mode for the row.
7. **The notes editor uses a SEPARATE `FocusRequester` from the title editor.** They live on different rows but a single `OutlineRow` can host both states (e.g. after committing title and switching to notes via the menu). Hold both in the row and trigger each via its own `LaunchedEffect`.
8. **The notes block has TWO render modes.** Editing → `BasicTextField`. Not editing AND notes are non-empty → read-only `Text` with `clickable { onBeginEditNotes() }`. Not editing AND notes empty → no render (the user uses the dropdown menu to start editing).
9. **`beginEditNotes(id)` works regardless of whether the node currently has notes.** When notes are empty, the editor opens with an empty buffer. When the user types and commits, `updateNotes` runs with whatever lines the user entered.
10. **Don't auto-commit notes on every keystroke.** The `notesBuffer` lives in the ViewModel and updates per keystroke (via `onNotesBufferChange`), but `commitEditInternal` only runs when triggered by another action (tap elsewhere, mutation op, save). This is intentional: it avoids running `TreeOps.updateNotes` on every character.
11. **`commitEditInternal` short-circuits if buffer matches current notes.** It compares `newNotes != node.notes` and only sets `dirty = true` if they differ. Otherwise it clears the editing state silently (no spurious dirty flag).
12. **Footgun: editing notes can corrupt structure.** A user typing `* fake heading` into the notes editor will, on next save+reopen, see that line become a real heading and split off any subsequent body lines. Same for breaking a drawer (e.g. removing `:END:`). M5 accepts this; documenting it in TODO.md as a future "validate before save" item.
13. **`OrgSerializer` already handles `Node.notes` round-trip.** The serializer emits notes via `emitLines(sb, node.notes)`, which appends each line + `\n`. As long as `updateNotes` puts the right list into `node.notes`, serialization is correct without any serializer change.
14. **Rotation preserves edit-in-progress.** Both `editing/editBuffer` and `editingNotes/notesBuffer` live in `OutlineState` inside the `OutlineViewModel`, which Android keeps alive across configuration changes. No extra `rememberSaveable` plumbing needed.
15. **`cycleTodo` calls `commitEditInternal` first.** Otherwise tapping a chip while a title edit is in progress would lose the edit. Same pattern as M4's mutators.
16. **Backward compatibility with the M4 `OutlineRow` signature.** Don't try to keep the old signature alive via default-argument tricks. Add the new params as required, update the call site at the `items {}` block. The whole file is in `private` scope so there are no external callers.

nix-shell --run "./gradlew test"

Manual on-device walkthrough:
1. Open `test.org`. Tap the `DONE` chip on the second heading → chip disappears (state null), strikethrough gone, `•` appears in the top bar.
2. Tap the same heading's title (where the chip was) → enters title-edit mode (no chip there to capture the tap). Cancel out by tapping another row.
3. Long-press that heading → menu shows "Toggle TODO". Tap → `TODO` chip appears.
4. Tap the new `TODO` chip → cycles to `DONE`. Tap again → cycles to null.
5. Tap the existing notes block under the first heading → it becomes a text field with the body lines pre-filled. Type a new line at the end (press Enter for a real newline). Tap another row → notes commit; block becomes read-only again with the new line included.
6. Long-press a heading with no notes → menu shows "Add notes". Tap → empty editor opens under the heading. Type "added body". Tap another row → notes block now renders "added body".
7. Tap the new notes block → editor reopens with "added body" pre-filled. Delete all content. Tap away → notes block disappears (empty list).
8. `Save`. `adb pull` and diff. The DONE removal shows as `* DONE Second…` → `* Second…`. Added/removed body lines match what you typed.
9. Edit notes, then on the host: `adb shell` append a line to the file. Tap `Save` → conflict dialog appears (M3 flow still works). Tap Overwrite → file gets your version including notes change.
10. Rotate the screen mid-notes-edit → buffer survives.

When all acceptance criteria pass, M5 is done.

## What M5 explicitly does NOT do

- Edit priority (`#A`/`#B`/`#C`) — M6/M7.
- Edit tags — M6/M7.
- Add a "+ TODO" placeholder chip — no, use the menu.
- Allow custom TODO workflow keywords — hardcode the cycle.
- Validate notes for accidental headings or broken drawers — known footgun.
- Single-row "notes-only" focus view — M6+ if at all.
- Multi-line title editing.
- Undo / redo.
- A Done button or any other explicit commit affordance for notes — tap-away is the only commit.
- Search inside notes.
- Persistence of in-progress edits across process death.
